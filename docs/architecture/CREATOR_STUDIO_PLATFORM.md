# Purpose

Creator Studio является платформой публикации контента проекта **My World**.

Документ описывает архитектуру платформы, её основные компоненты, принципы расширения и требования к добавлению новых типов контента.

Документ не описывает текущее состояние реализации проекта. Для этого используется `CURRENT_IMPLEMENTATION.md`.

---

# Architectural Goal

Главная цель Creator Studio — обеспечить единый механизм публикации любых доменных типов контента без дублирования инфраструктуры.

При добавлении нового типа контента должны изменяться только доменные компоненты.

Инфраструктура публикации должна переиспользоваться без изменений.

---

# Platform Principles

Creator Studio строится на следующих принципах.

## Domain Separation

Каждый тип контента имеет собственную:

* Markdown-модель;
* пользовательскую форму Author Mode;
* страницу отображения;
* Cloudflare endpoint.

Доменные модели не смешиваются между собой.

---

## Infrastructure Reuse

Общие механизмы платформы реализуются один раз.

Все домены используют их повторно.

---

## Minimal Domain Extension

При добавлении нового типа контента допускается изменение только доменной логики.

Платформенные компоненты изменяются исключительно при необходимости развития всей платформы.

---

## Consistent Author Experience

Авторский интерфейс должен вести себя одинаково независимо от публикуемого типа контента.

Это относится к:

* открытию Author Mode;
* публикации;
* Pending Publications;
* отображению ошибок;
* пользовательскому сценарию.

---

# Platform Components

В настоящий момент платформа включает следующие компоненты.

## Author Mode

Отвечает за:

* отображение интерфейса автора;
* модальные окна;
* формы публикации;
* проверку Studio Secret.

---

## Cloudflare Pages Functions

Отвечают за:

* приём запросов;
* валидацию;
* вызов Content Repository Layer.

---

## Content Repository Layer

Является центральным компонентом платформы.

Отвечает за:

* генерацию slug;
* генерацию Markdown;
* загрузку изображений;
* создание GitHub Commit;
* публикацию файлов.

Доменные endpoint не работают с GitHub напрямую.

Все операции проходят через Repository Layer.

---

## Upload Pipeline

Обеспечивает загрузку изображений.

Поддерживает:

* artwork image;
* character avatar;
* wall image.

Pipeline не зависит от конкретного типа контента.

---

## Markdown Generation

Каждый домен использует собственный шаблон Markdown.

Платформа отвечает за:

* формирование frontmatter;
* генерацию файлов;
* публикацию результата.

---

## Pending Publications

Отвечает за пользовательское состояние публикации.

Поддерживает:

* placeholder после публикации;
* sessionStorage;
* несколько одновременно ожидающих публикаций;
* автоматическое исчезновение после появления опубликованного контента;
* очистку устаревших записей.

---

## GitHub Publishing

Платформа самостоятельно:

* создаёт commit;
* публикует изменения;
* инициирует production deployment.

---

## Production Publishing Pipeline

Единый механизм публикации для всех доменных типов контента.

Pipeline:

```text
Author Mode
        │
        ▼
Cloudflare Pages Function
        │
        ▼
Content Repository Layer
        │
        ▼
GitHub Commit
        │
        ▼
Cloudflare Pages Deployment
        │
        ▼
Published Site
```

# Supported Content Types

На текущий момент платформой поддерживаются:

* Gallery
* Characters
* Wall

Каждый домен использует одну и ту же инфраструктуру публикации.

---

# Extension Points

При добавлении нового типа контента допускается создание:

* новой Markdown-модели;
* нового Repository Adapter (при необходимости);
* нового Cloudflare endpoint;
* нового Author Mode;
* нового публичного интерфейса.

При этом должны переиспользоваться:

* Content Repository Layer;
* Upload Pipeline;
* Pending Publications;
* GitHub Publishing;
* Production Publishing Pipeline.

---

# Platform Constraints

Creator Studio намеренно не является универсальной CMS.

Каждый новый домен проектируется отдельно.

Не допускается:

* универсальный редактор всех типов контента;
* единая динамическая форма публикации;
* отказ от доменных моделей.

---

# Architecture Validation

Архитектура платформы подтверждена практической реализацией следующих независимых доменов:

* Gallery
* Characters
* Wall

Реализация Characters Author Mode (Sprint 3) подтвердила возможность повторного использования платформенной инфраструктуры без изменения её архитектуры.

---

# Future Evolution

Дальнейшее развитие платформы предполагает добавление новых доменных типов контента через существующие точки расширения.

Изменение платформенной архитектуры допускается только при появлении требований, которые невозможно реализовать средствами существующих компонентов.

До появления таких требований предпочтительным подходом остаётся расширение платформы посредством новых доменных модулей при сохранении стабильности инфраструктурного ядра.
