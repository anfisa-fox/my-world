# DEVELOPMENT SESSION PROTOCOL

Version: 1.0

Status: Active

---

# Назначение документа

Данный документ определяет единый порядок начала, проведения и завершения инженерной сессии разработки проекта.

Его цель — обеспечить сохранение архитектурного контекста между различными ветками ChatGPT, исключить потерю ранее принятых решений и сделать процесс разработки воспроизводимым.

Настоящий документ описывает именно процесс разработки и не заменяет архитектурную или продуктовую документацию проекта.

---

# Основной принцип

Каждая новая ветка разработки рассматривается как продолжение существующего проекта, а не как новый проект.

Разработчик обязан восстановить контекст исключительно на основании приложенной документации и текущего состояния проекта.

Запрещается достраивать отсутствующий контекст предположениями.

При возникновении сомнений необходимо сначала задать вопрос или запросить просмотр существующей реализации.

---

# Документы, обязательные для начала разработки

Перед началом новой инженерной сессии должны быть приложены следующие документы:

```text
docs/

PROJECT_CONTEXT.md
CURRENT_IMPLEMENTATION.md
CURRENT_SPRINT.md

docs/handoff/
SPRINT_XXX_*.md
```

При наличии новых архитектурных документов они также должны учитываться.

---

# Этапы начала разработки

Перед началом реализации разработчик проходит три обязательных этапа.

## Шаг 1. Анализ

Изучить документацию проекта.

После анализа изложить понимание:

* текущего состояния проекта;
* реализованной функциональности;
* завершённых Sprint;
* архитектурных принципов;
* Scope текущего Sprint;
* известных ограничений;
* возможных рисков.

На данном этапе запрещается:

* писать код;
* предлагать изменения;
* предлагать рефакторинг.

---

## Шаг 2. Планирование

После анализа представить:

* понимание задач Sprint;
* последовательность реализации;
* предполагаемые файлы;
* порядок проверки результатов.

Это должен быть только план.

Реализация ещё не начинается.

---

## Шаг 3. Архитектурное подтверждение

После представления плана разработчик обязан остановиться и дождаться подтверждения.

До получения подтверждения запрещается:

* изменять код;
* предлагать новые архитектурные решения;
* изменять Scope Sprint;
* выполнять рефакторинг.

---

# Правила реализации

Во время разработки необходимо соблюдать следующие принципы.

## Один логический шаг

Один логический шаг должен соответствовать одному commit.

---

## Полная замена файлов

Все изменения выполняются только полной заменой файла.

Частичные patch не используются.

---

## Сначала анализ

Перед предложением изменений необходимо изучить существующую реализацию.

Запрещается проектировать новый механизм без анализа существующего кода.

Каждый шаг начинается с просмотра соответствующего файла.

---

## Переиспользование существующей архитектуры

Перед созданием новой реализации необходимо определить, существует ли аналогичный механизм в проекте.

Если существует проверенная реализация, она считается канонической и должна использоваться повторно.

Создание новой архитектуры допускается только после отдельного обсуждения.

---

## Scope Sprint

Разработчик не имеет права самостоятельно расширять Scope текущего Sprint.

Все дополнительные идеи:

* не реализуются;
* обсуждаются отдельно;
* при необходимости фиксируются в backlog или roadmap.

---

# Production First

Главной целью каждого Sprint является подтверждение полного production-сценария.

Приоритет всегда следующий:

```text
Реализация
        ↓
Production Verification
        ↓
UX
        ↓
Технические улучшения
```

Новые архитектурные идеи не должны откладывать подтверждение работы production-сценария.

---

# Канонические реализации

Если в проекте уже существует проверенная реализация аналогичной функциональности, она считается эталонной.

Новые разделы должны максимально повторять:

* архитектуру;
* пользовательский сценарий;
* структуру backend;
* подход к хранению данных;
* Production Pipeline.

Допускаются только изменения, обусловленные особенностями доменной модели нового раздела.

---

# Документирование

Каждый документ проекта имеет только одну ответственность.

Основные документы:

* `PROJECT_CONTEXT.md`
* `CURRENT_IMPLEMENTATION.md`
* `CURRENT_SPRINT.md`
* `docs/handoff/SPRINT_XXX_*.md`

Запрещается:

* собирать один Markdown-документ из нескольких сообщений;
* обновлять один документ по разделам;
* разрезать один документ на части.

Если документ перестаёт помещаться в один ответ ChatGPT, он должен быть разделён на несколько самостоятельных документов, каждый из которых имеет собственную ответственность.

Документы обновляются только полной заменой файла.

---

# Завершение Sprint

Sprint считается завершённым только после выполнения всех этапов:

1. Build.
2. Production Verification.
3. Architecture Audit.
4. Обновление `CURRENT_SPRINT.md`.
5. При необходимости обновление `CURRENT_IMPLEMENTATION.md`.
6. Создание `docs/handoff/SPRINT_XXX_*.md`.
7. Commit.
8. Push.
9. Только после этого допускается создание новой ветки разработки.

---

# Цель документа

Настоящий документ является обязательным для каждой новой инженерной сессии разработки.

Его соблюдение обеспечивает:

* сохранение архитектурного контекста;
* воспроизводимость процесса разработки;
* последовательное развитие проекта без потери ранее принятых решений;
* единый инженерный стандарт для всех последующих Sprint.
