# Назначение документа

Данный документ является официальной точкой входа в проект.

Его задача — дать любому новому разработчику, архитектору, аудитору или AI-помощнику быстрое и точное понимание проекта без необходимости изучать всю документацию.

Этот документ описывает только долгоживущий контекст проекта и практически не изменяется между спринтами.

Текущее состояние реализации описывается отдельно в `CURRENT_IMPLEMENTATION.md`.

Текущий этап разработки описывается отдельно в `CURRENT_SPRINT.md`.

При возникновении противоречий между текущими задачами и данным документом приоритет имеют утверждённые архитектурные решения проекта.

---

# Проект

**Personal Creative World**

Персональный творческий сайт для молодой художницы.

Это не портфолио, не блог и не универсальная CMS.

Главная идея проекта — создать живой цифровой мир, в котором автор чувствует себя хозяйкой и может самостоятельно развивать его без знания Git, GitHub или программирования.

Проект должен восприниматься как личное творческое пространство, а не как административная система управления контентом.

---

# Главная продуктовая гипотеза

Основная проверяемая гипотеза проекта:

> Автору должно хотеться регулярно возвращаться в свой мир и самостоятельно публиковать новый контент непосредственно внутри него.

Именно желание автора пользоваться своим миром является главным критерием успешности проекта.

Все технические решения должны служить этой цели.

---

# Архитектурные принципы

## Простота

Предпочтение всегда отдаётся самому простому решению, достаточному для решения текущей задачи.

Используется принцип Бритвы Оккама.

Любое усложнение архитектуры должно быть обосновано уже существующей потребностью, а не предполагаемым будущим развитием.

---

## Git — Source of Truth

Все опубликованные материалы хранятся исключительно в GitHub.

База данных отсутствует.

Markdown является каноническим форматом хранения контента.

GitHub Repository остаётся единственным источником истины.

---

## Static First

Основной сайт представляет собой статический экспорт Next.js.

Backend используется только там, где это действительно необходимо.

---

## Cloudflare Pages Functions

Вся серверная логика Creator Studio выполняется через Cloudflare Pages Functions.

Отдельный сервер приложения отсутствует.

---

## Author Mode

Автор никогда не покидает свой мир для управления им.

Author Mode является естественным продолжением сайта.

Проект сознательно избегает ощущения административной панели или корпоративной CMS.

# Порядок изучения проекта

Для быстрого погружения в проект рекомендуется следующий порядок.

1. Изучить данный документ (`PROJECT_CONTEXT.md`).
2. Ознакомиться с текущим состоянием реализации (`CURRENT_IMPLEMENTATION.md`).
3. Ознакомиться с текущим этапом разработки (`CURRENT_SPRINT.md`).
4. Только после этого переходить к специализированной документации соответствующего раздела.

# Роли AI

В проекте используются различные роли AI.

## Implementation Engineer

Реализует утверждённые решения.

Не изменяет архитектуру и продуктовую концепцию самостоятельно.

---

## Architecture Advisor

Проводит архитектурный аудит.

Помогает принимать технические решения.

Контролирует соблюдение утверждённых архитектурных принципов.

---

## Product Advisor

Помогает развивать продуктовую концепцию.

Проектирует пользовательский опыт.

Формирует новые пользовательские сценарии.

Не участвует непосредственно в реализации.

---

# Правила работы

Перед внесением любых изменений разработчик обязан:

1. Ознакомиться с `CURRENT_IMPLEMENTATION.md`.
2. Ознакомиться с `CURRENT_SPRINT.md`.
3. Подтвердить своё понимание текущего состояния проекта.
4. Назвать предполагаемые файлы, которые будут изменяться.
5. Объяснить выбранный подход.
6. Только после этого предлагать изменения.

# Принцип реализации

Во время активного Sprint запрещается самостоятельно расширять утверждённый Scope.

Любые идеи по улучшению:

* не реализуются сразу;
* обсуждаются отдельно;
* при необходимости фиксируются в продуктовой документации или roadmap.

Во время реализации действует принцип:

> **Implementation First.**

Сначала реализуется утверждённая концепция.

Только после пользовательской проверки допускается её изменение.

# Назначение документации

Документация проекта разделена по уровням ответственности.

**PROJECT_CONTEXT.md** отвечает на вопрос:

> Что это за проект и по каким принципам он развивается?

**CURRENT_IMPLEMENTATION.md** отвечает на вопрос:

> Как проект реализован сегодня?

**CURRENT_SPRINT.md** отвечает на вопрос:

> Что именно мы делаем сейчас?

Такое разделение позволяет безопасно переносить разработку между различными AI-помощниками, не теряя архитектурный и технический контекст проекта.
